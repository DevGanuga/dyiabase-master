Dyia production incident — resolved

May 14, 2026

What was wrong

The Round 4 deploy on May 3 added a unified subscription gate that reads subscription_tier from dyia_users. That column was supposed to exist already (migration 032_subscription_tier.sql) but Supabase never applied it because a second migration file shared the same 032_ prefix, and the runner only records one row per prefix in schema_migrations.

The result: every authenticated request to AI-powered routes — insights, assistant, smart quote pricing — failed at the SELECT, was caught by the route's user-lookup error branch, and returned { error: "User not found" }. Marco's mobile screenshot of “Insight unavailable · User not found” was literally that error rendered. The same root cause produced the “Unlock AI Insights” upsell for Pro users on web, because the client-side gate was also returning isPro=false for everyone.

What's fixed

  • The repair migration (038_subscription_tier_repair.sql) is committed to the repo and has been applied to production. The column is present, verified live.
  • Marco and Dev are now subscription_tier='pro'. They're back to full Pro on next page reload.
  • All AI routes succeed against the prod database again.

What's still loose

  • 30 paying users are sitting on tier='basic' in the database. The repair migration's column default filled every existing row with 'basic' before the backfill could write the correct value. Most of those rows are Hanna's QA test accounts that genuinely are on Basic. The ones that aren't — legitimate Pro subscribers from before Round 4 — will see Basic gating until their tier is corrected. The webhook self-heals on the next subscription event (renewal, plan change), so they'll naturally recover within a billing cycle. To fix them sooner, the bulk reconciliation script needs the real Stripe API key, which leads to the next item.
  • STRIPE_SECRET_KEY in .env.local contains a whsec_… value (webhook signing secret), not the actual sk_… API key. No Stripe API call from local can succeed against that. Worth double-checking the Vercel production env doesn't have the same swap.
  • Migration folder still has two files prefixed 032_. Same trap that caused this outage. One of them needs to be renumbered or deleted, plus a one-line CI check (ls migrations | awk -F_ '{print $1}' | sort | uniq -d) so this can never recur silently.
  • Brand-new-user forced paywall — product decision. Hanna's QA matrix lists “Brand-new user lands in app with FREE badge” as expected behavior. The app instead redirects them to the marketing site's pricing section. Round 4 documented this as deliberate but unresolved. Either the redirect comes out (matrix wins, free signups land on the locked-UI experience) or the matrix gets updated (paywall stays, that row removed from Hanna's test plan). Both are one-line changes; just need a call on which one ships.

Cause of the outage in one line

A migration that should have shipped in April was silently skipped by Supabase's runner because of a duplicate filename prefix, and Round 4 deployed code that depended on the column it would have added.
