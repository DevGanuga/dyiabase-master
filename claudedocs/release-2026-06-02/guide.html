Marco & the downgrade ticket

Customers can tip at Stripe checkout (100% to the merchant), and a Pro subscriber can be downgraded to Basic — by themselves at period end, or by an admin (scheduled or immediate).

Workstream A

Dyia Pay — Stripe Connect payments

Merchants collect card payments from their customers via a Stripe Connect destination charge. Dyia takes a flat 0.75% platform fee on the base amount only; money settles to the merchant's connected account.

Tips reconciled everywhere Expired-checkout cleanup Live hub refresh Stripe Connect Admin refunds

What changed this release

  • Customer tipping new — optional tip (15/18/20% or custom) on /pay/[token] across pay links, invoices, and quote/job payments. Tips go 100% to the merchant; the fee is on the base only, the tip is a separate Stripe line item.
  • Tip reconciliation in every path fix — the tip is computed from the paid Stripe total in the webhook, the customer verify route, and sync-pending. The sync-pending path previously marked paid without writing the tip, so tips under-reported when the webhook was dark. Now fixed.
  • Stuck-checkout cleanup new — a checkout.session.expired handler rolls an unpaid checkout_created row back to pending (still payable) and drops the dead session id, instead of leaving it stuck forever.
  • Live payments hub fix — the activity feed reconciles + refetches on tab focus/visibility (throttled 5s), so a payment made elsewhere appears without a manual reload.
  • Onboarding stays live — the account.updated webhook keeps the merchant's Connect flags current so the hub flips to "Live" automatically.
The "Get paid" hub — stats, activity feed, and Stripe Connect status.
Operational dependency: tip + paid reconciliation is correct in three layers, but the Stripe webhook must be configured correctly in production (see the Stripe section). verify and sync-pending are backups, not substitutes.
Workstream B

Subscriptions & billing

Three tiers gated by computeSubscriptionState(): basic (free), trial (14-day full Pro), pro (paid). The trial is full Pro and steps down to the chosen plan when it ends.

What changed this release

  • Self-service downgrade (Pro → Basic) new — POST /api/stripe/subscription/cancel sets Stripe cancel_at_period_end, mirrors the flag + subscription_ends_at in the DB (migration 040); DELETE undoes it. The user keeps Pro until the period they already paid for ends.
  • Admin cancel parity fix — /api/admin/cancel-subscription now writes cancel_at_period_end + end date optimistically on the scheduled path (previously waited on the webhook), matching the self-service route.
  • Admin plan switch — /api/admin/switch-plan does a live, prorated Stripe price swap for support cases ("downgrade me to Basic").
  • Dunning — invoice.payment_failed stamps a one-time grace window and emails the customer; cleared on recovery.
User-facing

Settings → Manage billing

Schedule a downgrade with an "undo," update the card via the Stripe portal, and see "Downgrading on <date>" without a Stripe round-trip.

Admin-facing

One place for the whole lifecycle

Grant Pro / trial, revoke, switch plan, and cancel (scheduled or immediate) — all on the user-detail page, with the scheduled state surfaced.

Workstream C

Full admin control

Two surfaces: the in-app AdminPanel (Sidebar → "Admin Panel") and the standalone /app/admin/* pages (dashboard, users, user detail, Dyia Pay, webhooks).

What changed this release

  • Dashboard crash fixed fix — /app/admin read the whole { metrics, users } payload as metrics, so every figure was undefined and the page crashed. Now unwrapped + guarded.
  • Full lifecycle on user-detail new — added Cancel-at-period-end / Cancel-immediately controls, surfaced the "Scheduled to cancel on <date>" state, and hardened load errors (403/500 no longer reads as "user not found").
  • Surfaces interconnected new — the in-app AdminPanel links to the previously URL-only tools (Dyia Pay, Webhooks, full dashboard); the Admin entry also appears in the mobile "More" sheet.
  • Auth aligned fix — /app/admin/layout now authorizes via is_admin or an admin/super_admin role — matching the APIs. A role-based admin is no longer bounced out of the standalone pages.
  • Webhook log fixed fix — migration 042 adds the error_message column the logger actually writes, so the admin Webhooks log stops under-reporting errors.
Workstream D

iOS-native app UI & motion

A platform-wide polish pass so the whole product (/app) shares one motion language and feels native on iPhone.

What changed this release

  • Unified motion system new — a tailwindcss-animate-compatible enter/exit engine with one iOS-spring curve. The animate-in / fade-in / slide-in-from-* / zoom-in-95 classes used across 8+ components were previously no-ops; they now resolve consistently. Full prefers-reduced-motion support.
  • iOS bottom tab bar new — the flat dark bar became translucent, blurred, theme-aware, and respects env(safe-area-inset-bottom) (no more home-indicator overlap), with an active-tab lift and press feedback.
  • Themed "More" sheet new — a proper iOS bottom sheet: grab handle, safe-area inset, theme-aware surface, springy slide, plus the Admin entry for admins.
  • Nothing clipped — content padding reserves the tab-bar height + safe area; overlays/sheets contain their overscroll (native rubber-band).
Verified on mobile (390×844): translucent tab bar + active state, "More" sheet animation, and no overflow or clipping on Dashboard, Payments, and Jobs.

Rollback

  • Code: git revert fb492ab (single commit) or redeploy the prior build.
  • Migrations are additive (new defaulted columns) — safe to leave even if code is rolled back; the columns are simply unused by older code.
  • No destructive schema changes. The only deletion is BetaBanner.tsx (replaced by TrialBanner).

Known gaps / follow-ups (not blockers)
