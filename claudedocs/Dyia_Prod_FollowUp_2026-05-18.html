Dyia — platform stability follow-up

May 18, 2026 · deployed to production

Context

After last week's subscription_tier incident note, Hanna and Marco hit four more issues during retest. Three of them were real bugs in our code, one was a synthetic-test-data limitation. Everything below is now live on dyia.io.

What shipped today

Symptom (what users saw) Fix
Brand-new signups were bounced from the app to the marketing pricing section. Hanna couldn't reach row 1 of the QA matrix and new founders never made it past the door. The forced redirect is gone. New accounts now land in the dashboard on the Free/Basic tier. The trial banner still drives the Pro upsell inline.
Marco's Jobs tab kept showing the “Unlock AI insights with Dyia Pro” upsell even though his account is Pro. The Jobs tab wasn't being told the user was Pro. Same prop-passing miss that <Quotes> already had fixed in Round 4. Both tabs now agree.
“Upgrade to Pro” clicks produced a generic error with no useful message. Hard to triage from logs. Every Stripe call now refuses to start when STRIPE_SECRET_KEY isn't a real API key, and returns a one-line, readable diagnostic instead of a blank 500. The user-facing copy is unchanged; the change is that we can see what's wrong the second it happens.
BUG-031 — customer pays via the Stripe link, merchant's dashboard still shows “Payment requested” hours later. New reconciliation route (/api/payments/sync-pending) asks Stripe directly about anything stuck in pending for the merchant and writes the truth back. The dashboard runs it on load and every time the merchant returns to the tab. The grid now self-heals even if a webhook is delayed, signature-mismatched, or unsubscribed in the Stripe dashboard.

What's still on QA / ops to confirm

  • Stripe sandbox subscriptions for Hanna's test accounts. The retest matrix needs each account in a specific subscription state. The QA accounts are now seeded directly in the database (+100 through +107, see QA matrix doc). She can validate badges, banners, and gating immediately. End-to-end Stripe flows (real checkout, real invoice history in the billing portal) still require a working sandbox checkout, which loops back to the production Stripe key being valid.
  • Verify the production Stripe env on Vercel. STRIPE_SECRET_KEY must begin with sk_test_ or sk_live_. STRIPE_WEBHOOK_SECRET must begin with whsec_. If those two are swapped in production (they currently are in .env.local), every Stripe API call fails — including the new sync-pending route — and users see opaque errors on checkout. With today's deploy, those failures now surface with a readable message in Vercel logs the moment they happen.
  • Bulk reconciliation of the ~30 pre-Round-4 Pro users still flagged tier='basic'. Unblocked the moment the production Stripe key above is confirmed correct. Until then, those users continue to self-heal on their next renewal or plan change event from Stripe.
  • Duplicate 032_ migration filename. Still in the repo. Recommend renumbering one of them and adding the one-line CI check from the previous note before the next migration ships, to prevent another silent-skip outage.
